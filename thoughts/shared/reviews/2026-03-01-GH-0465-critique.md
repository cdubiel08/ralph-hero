# Plan Critique: GH-465 Stacked Branch Strategy for Parallel Implementations

## Summary

The plan adds a stacked branch capability so that overlapping-file implementations can branch from their predecessor instead of `origin/main`. It touches four files: the `create-worktree.sh` script (new 3rd argument), `ralph-impl/SKILL.md` (base branch detection and rebase logic), `ralph-integrator.md` (rebase guidance for merged predecessors), and `ralph-team/SKILL.md` (metadata contract documentation). The scope is well-bounded and explicitly defers stream detection (GH-488) and multi-level cascade rebase.

## Completeness

**Verdict: Complete.**

All four changes are fully specified with exact file paths, line numbers, and code snippets. The plan covers the full lifecycle: worktree creation plumbing, implementation skill consumption, and integrator post-merge cleanup. Each change includes before/after context, making it unambiguous what to modify.

The plan correctly identifies that GH-488 owns stream detection in `ralph-team` and limits this plan to documenting the `base_branch` metadata contract -- the downstream consumer side. This separation is clean.

One minor gap: the plan does not mention updating the `remove-worktree.sh` script, but since stacked branches do not change worktree paths or cleanup semantics, this omission is intentional and correct.

The override logic in `create-worktree.sh` is sound: it tries `origin/$BASE_BRANCH_OVERRIDE` first (remote tracking), falls back to a bare ref, and ultimately falls back to the existing `BASE_BRANCH` with a warning. This graceful degradation handles the case where a predecessor branch has not been pushed yet.

The rebase logic in Step 6.3a correctly uses `git merge-base --is-ancestor` to detect whether the predecessor has been merged to main before rebasing -- this avoids unnecessary rebases when the predecessor is still in flight.

## Clarity

**Verdict: Clear. Success criteria are specific and testable.**

The seven success criteria (five checkbox verification items plus integration tests) cover:
- Positive case: override resolves to correct base branch
- Negative case: invalid override falls back gracefully
- Backward compatibility: no override preserves existing behavior
- Documentation: skill and agent files contain expected content

The three "automated" test cases in the success criteria are shell-invocation tests that can be run manually during implementation. The four "manual" criteria are document content checks. All are unambiguous.

One observation: the first automated test (`create-worktree.sh GH-test-stacked "" feature/main`) tests override resolution to `origin/main`, which is the same as the default. A more demonstrative test would use a non-main branch. However, this is a minor test design concern, not a plan defect -- the implementer can adjust the test branch name.

## Scope

**Verdict: Well-scoped. Exclusions are explicit and justified.**

The "What We're NOT Doing" section clearly excludes:
- Stream detection in `ralph-team` (deferred to GH-488)
- Full cascade rebase automation (multi-level stacks)
- External stacking tools (ghstack, Graphite)
- Runtime conflict monitoring

These are all reasonable deferrals. The plan is a single Small-sized issue with four focused file changes. The implementation approach (script plumbing first, then skill consumption, then documentation) provides a natural ordering where each step can be tested independently.

## Verdict

**APPROVED.** The plan is complete, feasible, clear, and well-scoped. All file references and line numbers verified against the current codebase. No blocking issues found.
