# Plan Critique: GH-509 — save_issue Iteration Param

## Verdict: APPROVED

Phase 2a of the iteration field support group plan is well-structured and ready for implementation.

## Analysis

### Completeness (Pass)

The plan covers all three code paths for the iteration parameter:

1. **Set by title or token** (`iteration: "Sprint 1"` or `iteration: "@current"`) -- resolves to `iterationId` via `resolveIterationId()` from Phase 1, then uses `value: { iterationId }` variant of `updateProjectV2ItemFieldValue`.
2. **Clear** (`iteration: null`) -- reuses existing `clearProjectV2ItemFieldValue` pattern already used by estimate/priority clearing.
3. **No-op** (`iteration: undefined`) -- preserves existing behavior, no mutation emitted.

Two files are identified for modification:
- [`plugin/ralph-hero/mcp-server/src/tools/batch-tools.ts`](https://github.com/cdubiel08/ralph-hero/blob/main/plugin/ralph-hero/mcp-server/src/tools/batch-tools.ts) -- `buildBatchMutationQuery` gets `valueType` discriminator
- [`plugin/ralph-hero/mcp-server/src/tools/issue-tools.ts`](https://github.com/cdubiel08/ralph-hero/blob/main/plugin/ralph-hero/mcp-server/src/tools/issue-tools.ts) -- `save_issue` input schema and handler logic

### Feasibility (Pass)

All referenced code locations verified against the codebase:

- `buildBatchMutationQuery` at `batch-tools.ts:86-126` confirmed -- currently hardcodes `value: { singleSelectOptionId: $optId }` at line 117.
- `save_issue` project field mutation block at `issue-tools.ts:1323-1425` confirmed -- uses the pattern of collecting `updates` array entries then calling `buildBatchMutationQuery`, with a separate loop for `fieldsToClear`.
- The clear mutation pattern at `issue-tools.ts:1410-1424` using `clearProjectV2ItemFieldValue` is confirmed and directly reusable.
- Existing set-or-clear patterns for `estimate` (lines 1367-1383) and `priority` (lines 1386-1402) provide exact templates to follow.
- `resolveIterationId()` dependency from Phase 1 (#508) is correctly identified.

### Clarity (Pass)

Four success criteria are specific and testable:

1. `save_issue({ number, iteration: "Sprint 1" })` sets iteration field via `value: { iterationId }`
2. `save_issue({ number, iteration: "@current" })` resolves via date math
3. `save_issue({ number, iteration: null })` clears via `clearProjectV2ItemFieldValue`
4. Regression: `save_issue({ number, workflowState: "In Progress" })` unchanged

All follow the `- [ ] Automated:` format with concrete assertions.

### Scope (Pass)

- Tightly bounded: only `save_issue` iteration param and `buildBatchMutationQuery` discriminator.
- "What we're NOT doing" section at the group level correctly excludes sprint lifecycle management, `@previous` token, and server-side filtering.
- "No direct output consumed by later phases" correctly characterizes this as a leaf consumer of Phase 1 infrastructure.

## Minor Notes (Non-Blocking)

1. **Backward compatibility of `buildBatchMutationQuery`**: The plan adds a `valueType` discriminator to the function signature. This changes the interface used by both `batch_update` tool (batch-tools.ts:496) and `save_issue` (issue-tools.ts:1406). The plan does not explicitly state backward compatibility strategy, but the natural implementation (defaulting `valueType` to `"singleSelectOptionId"` when omitted) is straightforward and low-risk. The implementer should ensure existing callers that do not pass `valueType` continue to work without modification.

2. **Iteration field name assumption**: The plan uses `"Sprint"` as the default iteration field name. The iteration field could have any user-defined name on a given project. The implementer should use `fieldCache.getFieldId()` with the actual field name (discovered via cache population in Phase 1) rather than hardcoding `"Sprint"`. The plan's use of `resolveIterationId(cache, projectNumber, fieldName, ...)` already accepts a `fieldName` parameter, so this is handled correctly.

## Conclusion

The plan is complete, technically sound, follows existing codebase patterns, and has clear success criteria. Ready for implementation.
