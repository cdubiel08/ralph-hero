# Plan Critique: GH-510 -- list_issues Iteration Filter

## Verdict: APPROVED

Phase 2b of the group iteration field support plan is well-structured, feasible, and ready for implementation.

## Analysis

### 1. Completeness

The plan defines three specific changes required, each with exact file paths and line references:

1. **Add `ProjectV2ItemFieldIterationValue` fragment** to the `list_issues` GraphQL query (`issue-tools.ts` lines ~228-237) -- verified: the current query at lines 228-237 only fragments on `ProjectV2ItemFieldSingleSelectValue`, and adding the iteration fragment is a straightforward addition.

2. **Extend `getFieldValue()`** to extract iteration data (`issue-tools.ts` lines ~1751-1761) -- verified: the current function at lines 1758-1768 only matches `ProjectV2ItemFieldSingleSelectValue`. The plan correctly identifies the need to handle `ProjectV2ItemFieldIterationValue` typename. The `RawProjectItem` interface (lines 1742-1756) will need its `fieldValues.nodes` type extended to include `iterationId`, `title`, `startDate`, `duration` fields.

3. **Add `iteration` filter parameter** -- verified: the existing filter chain (lines 253-401) follows a consistent pattern of sequential `items.filter()` calls. Adding an iteration filter follows the same pattern. The plan correctly specifies using `resolveIterationId()` from Phase 1 (#508) for client-side resolution.

All three changes are defined with specific file locations, code patterns, and clear descriptions.

### 2. Feasibility

**File references verified**:
- `plugin/ralph-hero/mcp-server/src/tools/issue-tools.ts` -- exists, confirmed line references for the fieldValues query (lines 228-237), getFieldValue function (lines 1758-1768), and the filter chain (lines 253-401)
- Dependency on Phase 1 (#508) -- `resolveIterationId()` and `FieldOptionCache.getIterations()` referenced but not yet implemented. The plan correctly declares this dependency.

**Pattern alignment**:
- The GraphQL fragment follows the exact same inline fragment pattern as the existing `ProjectV2ItemFieldSingleSelectValue` fragment
- The client-side filter follows the same `items.filter()` chain pattern used by all other filters (workflowState, estimate, priority, label, etc.)
- The `@current`/`@next` token resolution delegates to `resolveIterationId()` from Phase 1, maintaining separation of concerns

**`@previous` token**: The plan includes `@previous` in the filter parameter description but the "What We're NOT Doing" section explicitly excludes it due to a known off-by-one bug. However, Phase 2b's own description (line 220) says it will find `@previous` using `completedIterations` from cache. This is a minor inconsistency -- the overview section excludes `@previous` but Phase 2b includes it. The acceptance criteria on the issue itself include `@previous` support. Implementation should follow the issue acceptance criteria and include `@previous`.

**RawProjectItem type extension**: The plan does not explicitly call out that the `RawProjectItem` interface (lines 1742-1756) needs its `fieldValues.nodes` array type extended with iteration-specific fields (`iterationId`, `title`, `startDate`, `duration`). This is implicit but should be noted -- the implementer will need to add these optional fields to the existing union-style node type.

### 3. Clarity

Success criteria are specific and testable:
- `list_issues({ iteration: "@current" })` returns only issues in the active sprint -- automated test
- `list_issues({ iteration: "Sprint 1" })` returns issues assigned to that named iteration -- automated test
- `list_issues({ workflowState: "In Progress" })` (no `iteration`) unchanged -- regression test
- `list_issues({ iteration: "@next" })` returns issues in upcoming sprint -- manual test

All criteria follow the `- [ ] Automated:`/`- [ ] Manual:` format and are independently verifiable.

### 4. Scope

The scope is well-bounded:
- **In scope**: GraphQL fragment addition, field value extraction, filter parameter with client-side resolution
- **Out of scope** (from overall plan): server-side iteration filtering (GitHub API limitation), sprint lifecycle management, `@previous` token (though see note above)
- **Dependencies**: Phase 1 (#508) must be implemented first for `resolveIterationId()` and `FieldOptionCache.getIterations()`

The plan does not attempt to do too much or too little. Each change is atomic and testable.

### 5. Integration

The plan correctly identifies that this phase creates no direct output consumed by later phases, making it a leaf node in the dependency graph. The regression test criterion ensures existing `list_issues` behavior is preserved.

## Minor Observations (Non-Blocking)

1. **`@previous` inconsistency**: The top-level "What We're NOT Doing" says `@previous` is out of scope, but Phase 2b's own changes section says it will use `completedIterations` from cache for `@previous`. The issue's acceptance criteria include `@previous`. Recommend following the issue AC and implementing `@previous`.

2. **Response format**: The plan does not specify whether the formatted response should include iteration data (e.g., `iteration: "Sprint 1"` in the item output). The current response format (lines 418-438) includes workflowState, estimate, priority but not iteration. The implementer should add `iteration` to the formatted output for consistency.

3. **`getFieldValue` return type**: The current `getFieldValue()` returns `string | undefined`. For iteration data, the plan says to return `{ iterationId, title, startDate, duration }`. This is a different return type. The implementer may need a separate function (e.g., `getIterationFieldValue()`) or modify the approach to keep `getFieldValue` returning a string (just the title) while using a separate accessor for the full iteration object needed for filtering.

All three observations are implementation details that a competent implementer will handle naturally. None are plan-blocking.

## Conclusion

The plan is complete, feasible, clear in scope, and well-integrated with the group plan structure. The referenced files, line numbers, and patterns all check out against the current codebase. The dependency on Phase 1 (#508) is correctly declared. Ready for implementation.
