# Plan Critique: GH-468 Ralph Protocol Specs Phase 1 — Scaffold and Core Specs

## Verdict: APPROVED

The plan is well-structured, complete, and ready for implementation with no major issues.

## Assessment

### Completeness (Pass)

All five deliverables are clearly scoped with specific content requirements:
- `specs/README.md` — template, three-layer architecture, enablement convention, index
- `specs/artifact-metadata.md` — naming patterns, frontmatter schemas, comment protocol
- `specs/skill-io-contracts.md` — 15-skill contract table, stateless principle, result reporting
- `specs/skill-permissions.md` — permissions matrix, plugin overlay hooks
- `specs/agent-permissions.md` — tool whitelists, PreToolUse gates, stop gate keywords

Each phase has explicit "Changes Required" sections listing the exact content structure. The research document provides complete extracted data for all five specs.

### Feasibility (Pass)

All referenced source files verified as existing:
- `plugin/ralph-hero/hooks/scripts/ralph-command-contracts.json` — exists, contains all 7 command contracts
- `plugin/ralph-hero/hooks/hooks.json` — exists, contains plugin-level hook registrations
- `plugin/ralph-hero/hooks/scripts/ralph-state-machine.json` — exists
- All 3 agent `.md` files exist: `ralph-analyst.md`, `ralph-builder.md`, `ralph-integrator.md`
- All 15 skill directories with SKILL.md files exist

The data in the research document accurately reflects the actual codebase state. Spot-checked:
- `ralph-triage` SKILL.md confirms `allowed-tools: [Read, Glob, Grep, Bash, Task]` — matches research matrix
- `ralph-status` SKILL.md confirms no `allowed-tools` field — matches the documented gap
- `ralph-command-contracts.json` confirms the per-command contract structure with preconditions/postconditions

### Clarity (Pass)

Success criteria are specific and testable using the `- [ ] Automated:` / `- [ ] Manual:` format:
- Automated: `ls specs/*.md | wc -l` returns 5
- Automated: grep checks for `## Purpose`, `## Requirements`, `MUST` language
- Manual: enablement checkboxes match actual hook enforcement
- Manual: each spec is self-contained

### Scope (Pass)

"What We're NOT Doing" is explicit and well-bounded:
- Phase 2 (lifecycle/documents), Phase 3 (task/team), Phase 4 (fragments/refactor) are deferred
- No new hooks — specs document what IS and what SHOULD BE
- Not migrating conventions.md — designing from scratch

## Minor Observations (Non-blocking)

1. **Stream plan naming**: The plan lists `YYYY-MM-DD-stream-GH-{NNN}-{NNN}-{slug}.md` — the existing pattern uses non-padded `NNN` for stream plans vs padded `NNNN` for other artifact types. The spec should clarify whether stream plans also use zero-padding. This is a detail to resolve during implementation, not a plan deficiency.

2. **`ralph-impl` input states**: The research doc shows `ralph-impl` accepts `Plan in Review` and `In Progress` as input states. The plan correctly captures this in the skill contract table. Worth noting that `Plan in Review` as an input state for impl is because auto-review approval transitions directly to impl.

3. **Integration testing section**: The plan has a solid integration testing section with 5 checks. All are concrete and verifiable.

## Conclusion

Plan is comprehensive, well-researched, and implementation-ready. The 5-phase structure (README first, then specs in dependency order) is logical. All data sources are verified. Approve for implementation.
