# Plan Critique: GH-514, GH-515 — Skill workflowState Enforcement

## Summary

The plan proposes adding `workflowState: "Backlog"` to `save_issue` calls in two SKILL.md files (`form-idea` and `ralph-triage`) to fix issues being created without a workflow state. **Phase 1 (form-idea) is correct. Phase 2 (ralph-triage) will be blocked by the triage-state-gate hook.**

## Verdict: NEEDS_ITERATION

### Blocking Issue

**Phase 2 will fail at runtime.** The `triage-state-gate.sh` hook (PostToolUse on `ralph_hero__save_issue`) validates that any `workflowState` set during triage is in the allowed list:

```bash
# triage-state-gate.sh:22
valid_output="${RALPH_VALID_OUTPUT_STATES:-Research Needed,Ready for Plan,Done,Canceled,Human Needed}"
```

**"Backlog" is NOT in this list.** When the SPLIT path calls `save_issue` with `workflowState: "Backlog"`, the hook will block with:

```
Invalid triage state transition
Command: triage
Attempted state: Backlog
Valid output states: Research Needed,Ready for Plan,Done,Canceled,Human Needed
```

The plan explicitly claims (line 43): "Not adding hooks or validation — existing hooks already accept 'Backlog' as a valid state." **This claim is false for ralph-triage.**

### Fix Options for Phase 2

1. **Add "Backlog" to valid output states** — Update the `set-skill-env.sh` call in ralph-triage's SessionStart hook to include `RALPH_VALID_OUTPUT_STATES="Research Needed,Ready for Plan,Done,Canceled,Human Needed,Backlog"`. This is the cleanest fix since triage's SPLIT path legitimately creates new issues that should start in Backlog.

2. **Skip workflowState in triage SPLIT path** — Don't add `workflowState` to the SPLIT path at all; instead, rely on GH-516 (`create_issue` Status sync) to handle this at the MCP level. But this leaves the gap open until GH-516 ships.

3. **Use `__COMPLETE__` token** — Match ralph-split's pattern (`workflowState: "__COMPLETE__"`, `command: "ralph_triage"`). But this is more complex and the `__COMPLETE__` resolution logic may not map to "Backlog" for the triage command.

**Recommendation**: Option 1. Add "Backlog" to valid output states for triage. It's a one-line change to the skill frontmatter and correctly models that triage's SPLIT path creates new issues in Backlog.

### Pattern Claim Verification

The plan states: "ralph-split correctly sets `workflowState: 'Backlog'` on created sub-issues."

**This is inaccurate.** ralph-split actually uses a two-step pattern:
- Step 6.4: `workflowState: "__COMPLETE__"` with `command: "ralph_split"` (lines 223-229)
- Step 9: Separately sets the appropriate state ("Ready for Plan", "Research Needed", or "Backlog") with `command: "ralph_split"` (lines 313-325)

ralph-split never literally sets `workflowState: "Backlog"` as a hardcoded value. The `__COMPLETE__` token is resolved by the command's state machine. This doesn't affect the correctness of the proposed fix, but the reasoning justification is inaccurate.
