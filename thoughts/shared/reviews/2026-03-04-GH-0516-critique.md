### `save_issue` pattern ⚠️ Factual discrepancy
The plan states: *"The `save_issue` handler calls `syncStatusField()` after every workflowState change"* — this is **incorrect**.

`save_issue` does **inline** status sync at lines 1343-1354 using `WORKFLOW_STATE_TO_STATUS` directly within an aliased batch mutation. It does NOT call the `syncStatusField()` helper. The `syncStatusField()` helper is only called in `relationship-tools.ts` (by `advance_issue` at lines 626 and 861).

**Impact on plan:** None. The proposed fix (using `syncStatusField()`) is still correct. The helper exists precisely for this use case — it's simpler than duplicating the inline batch pattern, and achieves the same result. The existing codebase already has both patterns (`save_issue` inline, `advance_issue` via helper), so using the helper in `create_issue` is a valid choice.
