# Critique: GH-732 Hero Orchestrator Agent() to Skill() Dispatch Migration

## Verdict: NEEDS_ITERATION

## Overall Assessment

The plan is well-structured, technically accurate in its codebase references, and correctly scoped for the hero orchestrator migration. All line numbers in `hero/SKILL.md` were verified against the actual file and are exact. The Skill() argument formats match each skill's `argument-hint:` field. The plan correctly identifies and excludes cross-repo general-purpose Agent() calls from migration. However, there is one major discrepancy with the issue's acceptance criteria: the issue explicitly requires updating `hello/SKILL.md` dispatch table, but the plan declares it out of scope. This must be resolved before implementation can proceed.

## Issues Found

### MAJOR: hello/SKILL.md excluded despite explicit issue requirement

**Severity**: MAJOR

The issue body states under Scope: "Modify `hello/SKILL.md` dispatch table similarly." The acceptance criteria include: "Hello skill dispatch table updated to match." However, the plan's "What We're NOT Doing" section explicitly says "Modifying hello/SKILL.md -- hello runs inline (user-invocable), so its Agent() dispatch already works correctly."

The plan's rationale is partially valid -- hello runs with `context: inline` and its Agent() calls spawn isolated sub-agents that preload skills via their agent definitions. These sub-agents don't need to dispatch further sub-agents (triage doesn't spawn sub-agents, merge doesn't spawn sub-agents, etc.), so the "Agent() can't nest Agent()" limitation is less relevant for hello's dispatch patterns.

However, this directly contradicts the issue's acceptance criteria. The plan must either:
1. Add a phase to migrate hello's 6 Agent() dispatch patterns (lines 117-122) to Skill() calls, OR
2. Explicitly document why hello is excluded with a note that acceptance criterion "Hello skill dispatch table updated to match" should be removed from the issue, and get human confirmation before proceeding.

**Affected lines in hello/SKILL.md**: 117-122 (triage-agent, review-agent, merge-agent, research-agent, plan-agent, triage-agent for backlog).

### MINOR: PR dispatch line reference may be inaccurate

**Severity**: MINOR

The plan references "Lines 378-379" for PR dispatch, but the current file at lines 378-379 reads:
```
#### PR tasks
After all implementations complete, report all issue numbers with PR URLs and "In Review" status.
```

This is correct content but it's not an Agent() call -- it's a text instruction. The plan proposes adding an explicit `Skill("ralph-hero:ralph-pr", args="#NNN")` call here, which is an enhancement rather than a migration. The plan should clarify this is a new addition, not a replacement of an existing Agent() call. The distinction matters for reviewers assessing the change's scope.

### MINOR: Missing parallelism semantics clarification

**Severity**: MINOR

The current execution loop (line 241) says "Execute all unblocked tasks simultaneously (multiple Agent() calls in a single message)." The plan's replacement (section 1) changes this to describe Skill() dispatch but does not explicitly address whether multiple Skill() calls can be made in parallel within a single turn. Skill() runs inline and is blocking -- if the hero needs to dispatch multiple research tasks simultaneously, can it issue multiple Skill() calls in one message? The plan should clarify whether parallel Skill() dispatch is possible or whether the execution loop now becomes sequential per-task.

### MINOR: Hook mechanism difference not fully explored

**Severity**: MINOR

The plan notes that skill hooks fire via `RALPH_COMMAND` (SessionStart) when invoked via Skill(), while agents fire hooks via `agent_type` discrimination. The "Dispatch Architecture" replacement section mentions this but doesn't address whether any existing hooks in `hooks.json` discriminate on `agent_type` and would need `RALPH_COMMAND` fallback paths. If hooks currently use `agent_type === "impl-agent"` to gate behavior, those hooks won't fire when hero dispatches via Skill() instead. The plan should verify no hook enforcement is lost in the migration.
