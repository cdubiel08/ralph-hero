# Critique: GH-377 Complete Autonomous Loop

## Verdict: NEEDS ITERATION

The plan demonstrates strong research into the current system state and is well-structured across four phases. However, it has several critical gaps that would cause the implementation to fail at runtime. The most serious issue is that `ralph-val`, `ralph-pr`, and `ralph-merge` all require an issue number argument and have no queue-picking logic, yet the plan wires them into `ralph-loop.sh` identically to skills like `ralph-triage` and `ralph-impl` that do self-select from queues.

## Issue 1 (Critical): `ralph-val`, `ralph-pr`, and `ralph-merge` lack queue-picking logic

The plan wires these skills into the loop with the same `run_claude "/ralph-hero:ralph-val"` pattern used by `ralph-triage` and `ralph-impl`. However:

- **`ralph-val`** (SKILL.md Step 1): Requires `<issue-number>` as argument. Has no fallback queue-selection logic. Has no "Queue empty" output path.
- **`ralph-pr`** (SKILL.md Step 1): Requires `<issue-number>` as argument. Has no fallback queue-selection logic. Has no "Queue empty" output path.
- **`ralph-merge`** (SKILL.md Step 1): Requires `<issue-number>` as argument. Has no fallback queue-selection logic. Has no "Queue empty" output path.

When invoked without an issue number from the loop, these skills will fail immediately (they have no "If no issue number" branch like `ralph-impl` does).

**Impact**: The integrator phases will never work as written. The loop cannot invoke these skills without knowing which issue to operate on.

**Recommendation**: The plan must either:
1. Add queue-picking logic to each of these three skills (find the highest-priority XS/S issue in the appropriate workflow state), OR
2. Add a pre-dispatch step in `ralph-loop.sh` that queries for the next eligible issue number and passes it to the skill invocation.

Option 1 is more consistent with how the existing skills work (`ralph-triage`, `ralph-impl`, `ralph-research`, `ralph-plan`, `ralph-review` all self-select). This is a significant scope addition that may push the estimate beyond M.

## Issue 2 (Critical): `ralph-hygiene` does not output "Queue empty"

The plan (Phase 1, item 2) says to verify that `ralph-hygiene` outputs "Queue empty" when there is nothing to do. Actual inspection shows:

- `ralph-hygiene/SKILL.md` outputs `"Hygiene complete."` at line 96 -- never "Queue empty".
- Furthermore, hygiene is a non-queue skill: it always runs, scans the board, and reports. There is no concept of "no work to do" for hygiene -- it always produces an archive eligibility report.

The current loop already handles hygiene differently: it calls `run_claude` but does not check its return value for queue-empty (line 149: `run_claude "/ralph-hero:ralph-hygiene" "hygiene"` -- no `if` wrapper, just direct invocation, and `work_done=true` is set unconditionally).

**Impact**: Low -- hygiene works fine as-is. But the plan's instruction to "verify/edit queue-empty text" for hygiene is wrong. It should note that hygiene is always-run and not subject to queue-empty detection.

## Issue 3 (Moderate): `code-review:code-review` does not "auto-detect the PR from gh context"

The plan states at line 44: "`code-review:code-review` auto-detects PR from `gh` context". However, inspection of the actual code-review plugin command definition shows:

- The plugin description says "Provide a code review for the given pull request" -- it expects a PR to be "given".
- The plugin uses `gh pr diff`, `gh pr view`, `gh pr comment`, etc. -- these commands need to be run from within a repo context where `gh` can resolve the PR.
- The plugin does NOT have an explicit auto-detection mechanism. It works when `gh` commands can resolve the current branch to a PR.

The plan's `ralph-code-review` skill (Phase 2, Step 4) invokes `Skill("code-review:code-review")` without any arguments. For this to work, the skill must be running from a worktree where the current branch corresponds to the PR.

The plan does address this partially: it checks for the worktree at `$GIT_ROOT/worktrees/GH-NNN` and `cd`s into it. But the `Skill()` invocation does not pass the PR number, and the code-review plugin's behavior when no PR number is given depends on `gh` resolving the current branch. This will work IF the worktree is checked out on the correct branch, but the plan should explicitly verify that `gh pr view --json number` resolves successfully before invoking the code-review skill.

**Recommendation**: Add a pre-check step in `ralph-code-review` Step 4 that verifies `gh pr view --json number` returns the expected PR before invoking the Skill.

## Issue 4 (Moderate): Three-round address mode failure path is incomplete

The plan says (Phase 2, Step 5-6): after 3 rounds of review+fix, if issues remain, post a comment and STOP. However:

- The issue's workflow state remains "In Review".
- On the next loop iteration, `ralph-code-review` will pick the same issue again (it is still "In Review" with an open PR).
- This creates an infinite retry loop across loop iterations, burning budget.

**Recommendation**: After 3 rounds, the skill should either:
1. Move the issue to "Human Needed" to take it out of the code-review queue, OR
2. Add a label (e.g., "review-exhausted") and check for that label before picking issues, OR
3. The loop should track which issues have been attempted in the current iteration and skip them.

## Issue 5 (Minor): `ralph-code-review` uses `Agent()` with `subagent_type="ralph-hero:impl-agent"` but `Skill()` for code-review

The plan mixes orchestration patterns in the new `ralph-code-review` skill:
- Step 4 uses `Skill("code-review:code-review")` -- invokes a plugin skill.
- Step 5 uses `Agent(subagent_type="ralph-hero:impl-agent", ...)` -- invokes an agent.

This is fine architecturally, but the `impl-agent.md` currently has `model: opus` (most expensive model). Running impl-agent just for address mode is heavy. The plan should note this cost implication in the Performance Considerations section. A code review + 3 rounds of opus address mode could easily exceed the $3 budget set for the `code-review` justfile recipe.

---

## Issue 6 (Minor): Justfile `loop` recipe default change is not called out as breaking

The plan changes the `loop` recipe's `review` parameter default from `"skip"` to `"auto"` (Phase 3, item 4). While the Migration Notes mention this, the justfile change is buried in Phase 3 rather than Phase 1 where the equivalent script change is made.

It would be cleaner to make both the script default and the justfile default change in the same phase to avoid inconsistency during partial implementation.

---

## Issue 7 (Minor): Missing `ralph-code-review` agent in `allowed-tools`

The `ralph-code-review` SKILL.md includes `Skill` in `allowed-tools`, which is needed to invoke `code-review:code-review`. However, it also includes `Agent` for invoking `impl-agent`. The `impl-agent.md` definition references the `ralph-hero:ralph-impl` skill. For the nested invocation chain (`ralph-code-review` -> `Agent(impl-agent)` -> `ralph-impl`) to work, the impl-agent needs access to all the tools that `ralph-impl` requires (Write, Edit, etc.). The plan's agent definition for `code-review-agent.md` does not include Write or Edit, but the impl-agent.md already does. This should work since the impl-agent has its own tool permissions, but the plan should verify this nested tool permission chain explicitly.

## Scope Assessment

The plan is estimated as M. Given the issues above:

- Adding queue-picking logic to `ralph-val`, `ralph-pr`, and `ralph-merge` (Issue 1) is significant additional scope. Each skill needs a new "If no issue number" branch with list_issues queries, plus "Queue empty" output.
- The 3-round failure path (Issue 4) needs a state transition to prevent infinite retry.

These additions push the effective scope toward the upper end of M. The plan should acknowledge this and consider whether the queue-picking changes should be a prerequisite issue.

## What the Plan Gets Right

1. **Research quality is strong**: The current state analysis accurately identifies all bugs and missing pieces. Line references are almost exactly correct.
2. **Phase ordering is logical**: Fix fundamentals first, then add new skill, then wire into loop, then make merge headless.
3. **Backwards compatibility is well-considered**: The `--review=skip` escape hatch and `--auto-merge` opt-in are good design decisions.
4. **Success criteria are specific and testable**: Each phase has concrete grep-based verification commands.
5. **The "What we're NOT doing" section is well-scoped**: It explicitly excludes parallel processing, budget aggregation, and CI triggering.
