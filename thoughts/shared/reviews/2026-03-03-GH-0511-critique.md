# Plan Critique: GH-511 setup_project Iteration Field Creation (Phase 2c)

## Verdict: APPROVED

## Dimensions

### 1. Completeness

The plan defines exactly two changes with clear code snippets:
- A new `createIterationField()` function in `project-tools.ts` using `createProjectV2Field` with `dataType: ITERATION`
- A new `createIterationField?: boolean` parameter on `setup_project` (default `false`)

Both are described with function signatures, GraphQL mutations, and integration points. Success criteria cover automated structural tests and manual verification.

### 2. Feasibility

All referenced files verified:
- [`project-tools.ts`](https://github.com/cdubiel08/ralph-hero/blob/main/plugin/ralph-hero/mcp-server/src/tools/project-tools.ts) -- 683 lines. `createSingleSelectField()` (lines 585-633) provides the exact pattern to follow. The blank path in `setup_project` (lines 310-365) shows the insertion point.
- [`setup-project-template.test.ts`](https://github.com/cdubiel08/ralph-hero/blob/main/plugin/ralph-hero/mcp-server/src/__tests__/setup-project-template.test.ts) -- uses structural (source-reading) assertion pattern, consistent with plan's test approach.
- Research document confirms `createProjectV2Field(dataType: ITERATION, iterationConfiguration: {...})` is fully supported by GitHub's GraphQL API.

Phase independence is correctly justified: `setup_project` only creates the field via mutation and does not need Phase 1's cache/fragment infrastructure. The `ensureFieldCacheForNewProject` call after creation will not store iteration option IDs (no iteration fragment in `fetchProjectForCache` yet), but this is harmless since #511 does not consume iteration IDs.

### 3. Clarity

Success criteria are specific and testable:
- `setup_project({ createIterationField: true })` creates a "Sprint" field
- `setup_project()` without flag creates only 3 existing fields (backward compat)
- Manual: Sprint field visible in GitHub UI

Default values are explicit: name="Sprint", duration=14 days, startDate=next Monday (computed at call time).

### 4. Scope

Well-bounded. No sprint lifecycle management, no sprint reports, no cache read path. The plan states "No direct output consumed by later phases" confirming clean isolation.

## Minor Observations (Not Blockers)

1. No explicit error handling for duplicate "Sprint" field creation. Acceptable for XS scope -- GitHub API error will surface through existing catch block at lines 405-408.
2. The `iterationConfiguration` input shape (with top-level `duration`/`startDate` plus `iterations[]` array) matches the research findings and GitHub API documentation.
3. The helper function correctly separates concerns: `createIterationField()` takes `startDate` as input, `setup_project` computes "next Monday" as the caller.

## Issues

None. Plan is ready for implementation.
