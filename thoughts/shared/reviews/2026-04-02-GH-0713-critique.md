## Test Coverage Assessment

### Test 1: Plugin skill namespace resolution — WELL DEFINED

The test correctly verifies that backtick preprocessing actually resolves env vars at load time (the core Phase 2 contract). Acceptance criterion is specific: `Owner: cdubiel08` in preloaded content, not the literal `$RALPH_GH_OWNER` string. This is the right oracle.

### Test 2: Per-phase agent model verification — ADEQUATE

Tests one agent per model tier (sonnet/opus/haiku) rather than all 10 agents. This is the correct tradeoff for a manual test. Covering `research-agent`, `impl-agent`, and `pr-agent` gives confidence across all three tiers without requiring 10 separate invocations.

**Minor gap**: Model tier verification via Claude Code's UI is indirect — the agent doesn't self-report its model; you observe behavior. The plan accepts "confirms sonnet model" without specifying the observation mechanism (e.g., cost indicator, `--debug` output, agent response latency). This is a documentation ambiguity, not a blocking issue, since the tester knows their tooling.

### Test 3: Hook agent_type discrimination — SPECIFIC AND TESTABLE

Tests both routing paths: `impl-agent` → `impl-plan-required.sh`, and `research-agent` → `branch-gate.sh`. These are the two critical dispatch cases in `agent-phase-gate.sh`. The acceptance criteria match the actual case statement in the script.

**Note**: The `pr-agent`, `merge-agent`, and `val-agent` integrator tier agents do NOT appear in `agent-phase-gate.sh`'s case statement — they fall through to `allow`. This is intentional (integrators are not restricted by phase-gate hooks), and Test 3 does not need to cover them.

### Test 4: Hero end-to-end pipeline — APPROPRIATELY SCOPED

"Through at least the research phase" is the right stopping point for a manual test that avoids side effects (Linear state changes, worktree creation, PR creation). The five sub-criteria (correct agent dispatched, correct model, MCP tools work without params, hooks fire, artifact path passes naturally) together constitute full validation of the architecture's core contracts.

**Known risk acknowledged**: The issue body already flags "artifact path passing via natural language prompts" as a monitoring point. The plan is honest about this being the trickiest part of the new dispatch model.

### Test 5: Direct skill backward compatibility — CRITICAL PATH

This test validates the `RALPH_COMMAND` guard in `agent-phase-gate.sh` — the "double-blocking" prevention mechanism. Without this test, a regression where direct skill invocation is blocked by `agent-phase-gate.sh` could go undetected. Well-placed and specific.

### Test 6: Cross-repo env var resolution — CORRECTLY SPECIFIED

Tests that `RALPH_GH_REPO` resolves to the target repo (`ralph-engine`) not the plugin's home repo (`ralph-hero`). The specific expected values (`Owner: cdubiel08`, `Repo: ralph-engine`, `Project: 7`) make this a concrete, unambiguous pass/fail test.

**Path note**: The plan uses `/Users/dubiel/projects/ralph-engine` (macOS path). The environment is Linux/WSL (`/home/chad_a_dubiel/`). The tester will need to adjust the path, but this is a documentation detail, not a plan deficiency.

## Warnings (Non-Blocking)

### 1. README.md Has Stale Agents File Tree

`plugin/ralph-hero/README.md` lines 200–202 still show the old file tree listing `ralph-analyst.md`, `ralph-builder.md`, `ralph-integrator.md`. These files were deleted in Phase 5. The README file tree is not covered by Phase 6's scope (which targeted `CLAUDE.md` and `specs/`). This won't affect runtime behavior but creates misleading documentation.

**Recommendation**: Update README.md agents section to list the 10 per-phase agents. This can be addressed as a follow-up or squashed into the PR before merge.

### 2. Hero Dispatch Does Not Explicitly Use pr-agent/merge-agent/val-agent

The hero `SKILL.md` implements the "PR tasks" section as prose ("After all implementations complete, report all issue numbers with PR URLs and 'In Review' status") rather than explicit `Agent(subagent_type="ralph-hero:pr-agent", ...)` dispatch calls. The integrator-tier agents (`pr-agent`, `merge-agent`, `val-agent`) exist but are not wired into hero's dispatch table — hero relies on `RALPH_COMMAND` for PR creation instead.

This is consistent with the current architecture (hero itself handles PR reporting), but the CLAUDE.md table implies these agents are dispatched by hero. The mismatch is a documentation ambiguity, not a functional bug. Test 4 will surface this during execution.

### 3. Model Verification Mechanism Unspecified

Test 2 says "confirms sonnet/opus/haiku model" but doesn't say how. Options include: (a) ask the agent what model it's using, (b) observe Claude Code's model indicator in the UI, (c) check `--debug` output. All three work; testers should document which mechanism they used in their PASS comment.
