# Plan Critique: GH-719 relationships Table Schema Migration

## Verdict: APPROVED

The plan is well-researched, correctly scoped, and follows the existing migration pattern in the codebase. The table rebuild approach is the right choice because SQLite cannot alter CHECK constraints via `ALTER TABLE`. The probe query detection mechanism is sound, the SQL is correct, and the proposed tests are thorough. One minor issue identified (async test syntax) is non-blocking and easily fixed during implementation.

---

## Technical Verification

### 1. Probe Query Approach -- CORRECT

The plan uses `this.db.prepare("SELECT context FROM relationships LIMIT 0").get()` to detect whether the `context` column exists. In `better-sqlite3`, `.prepare()` compiles the SQL statement at prepare time -- if the column does not exist, the prepare call itself throws `SqliteError: no such column: context`. This means the probe reliably triggers the catch block on old schemas and succeeds silently on new schemas. The `LIMIT 0` ensures zero rows are scanned, making it effectively free.

### 2. Table Rebuild SQL -- CORRECT

The four-statement sequence is the standard SQLite table rebuild pattern:

```sql
CREATE TABLE relationships_new (...);
INSERT INTO relationships_new (...) SELECT ... FROM relationships;
DROP TABLE relationships;
ALTER TABLE relationships_new RENAME TO relationships;
```

Verified specifics:
- The new table definition matches the code-defined schema at `db.ts:120-126` exactly (both FKs, expanded CHECK, context column)
- The `INSERT` correctly omits `context` from the column list and select list, so migrated rows get `NULL` context -- this is correct behavior
- The `DROP TABLE` will cascade-drop the existing `idx_rel_target` index, and the subsequent `CREATE INDEX IF NOT EXISTS idx_rel_target` recreates it

### 3. Index Recreation -- CORRECT

The plan explicitly includes `CREATE INDEX IF NOT EXISTS idx_rel_target ON relationships(target_id, type)` after the rename. This is necessary because `DROP TABLE relationships` destroys the original index. Without this line, queries filtering on `target_id` would degrade to full table scans. The plan handles this correctly.

### 4. Empty Table Edge Case -- HANDLED

If the `relationships` table has zero rows, `INSERT INTO relationships_new ... SELECT ... FROM relationships` is a no-op, which is the correct behavior. The table structure is still rebuilt with the new schema. No issue here.

### 5. Placement After is_stub Migration -- CORRECT

The plan places the new migration after line 167 (the `is_stub` migration). This is the right location -- it runs after `CREATE TABLE IF NOT EXISTS` establishes the tables for fresh databases and after the `is_stub` migration handles that column. Migration order is correct.

## Test Coverage Assessment

### Test 1: Old schema rebuild -- WELL DEFINED

Creates a raw database with the old schema (no `context`, narrow CHECK, no `target_id` FK), inserts test data, opens via `KnowledgeDB`, and verifies:
- Existing relationship data preserved with `context: null`
- New context column is writable and readable
- New CHECK types (`post_mortem`, `untyped`) are accepted

The raw database setup correctly includes all tables (`documents`, `tags`, `relationships`, `outcome_events`, `sync`) and the `is_stub` column on `documents`, preventing the `is_stub` migration from interfering. This is thorough.

### Test 2: Idempotency -- CORRECT

Opens a fresh `KnowledgeDB`, writes data with context, closes, reopens, and verifies data integrity. This confirms the probe query succeeds when the column exists and no rebuild occurs.

### Test 3: Traverse after migration -- VALIDATES THE FIX

This test directly validates the reported bug: `knowledge_traverse` failing with "no such column: r.context" on migrated databases. It creates an old-schema database, opens it via `KnowledgeDB`, and calls `traverser.traverse()`.

**Minor issue**: The test uses `const { Traverser } = await import("../traverse.js")` inside a synchronous `it()` callback. The existing test file has zero `async` test callbacks. This needs `it("traverse works after migration", async () => {` to work correctly. Without the `async` keyword, the `await` expression will cause a TypeScript compilation error. This is trivially fixable during implementation and does not affect the plan's validity.

## Scope Assessment

### "What We're NOT Doing" -- WELL DEFINED

The three exclusions are all reasonable:

1. **No reindex trigger** -- Correct. Migrated rows having `context = NULL` is acceptable; context is populated on next reindex via `parser.ts` context extraction. Forcing a reindex would be heavy-handed for a migration.

2. **No explicit FK enforcement concern** -- Correct. `better-sqlite3` enables `PRAGMA foreign_keys = ON` by default, but the FK on `target_id` pointing to `documents(id)` will not cause issues because existing `target_id` values that do not match a `documents` row were already in the table before migration. SQLite foreign key enforcement only applies to new INSERT/UPDATE operations, not to data that was already present when the FK was defined via table rebuild.

3. **No version tracking / migration framework** -- Correct for the project's scale. The try/catch probe pattern is simple, idempotent, and matches the existing `is_stub` migration convention. Introducing a migration framework for two migrations would be over-engineering.

## Transaction Safety (Non-Blocking Warning)

The four-statement rebuild runs via `this.db` without an explicit `BEGIN`/`COMMIT` transaction wrapper. If the process crashes between `DROP TABLE relationships` and `ALTER TABLE relationships_new RENAME TO relationships`, the database would be left with `relationships_new` but no `relationships` table.

However:
- The existing `is_stub` migration also does not use explicit transactions
- SQLite's WAL mode provides crash safety for individual statements
- The four statements run in microseconds on a small table
- A follow-up migration could detect and recover from this state (though this is extremely unlikely)

This is a theoretical risk, not a practical one. Not blocking.

## Issues Found

### 1. Async test callback (Minor, Non-Blocking)

The third test ("traverse works after migration") uses `await import()` inside a synchronous `it()` callback. It needs `async` added to the callback: `it("traverse works after migration", async () => {`. This is a one-word fix during implementation. Alternatively, the Traverser import could be moved to a top-level import like the existing `traverse.test.ts` does, avoiding the async requirement entirely.
