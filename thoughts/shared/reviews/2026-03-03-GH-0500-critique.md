## Minor Observations (Non-Blocking)

### 1. warn() exits immediately

The plan notes some ambiguity about using `warn()` vs `echo >&2` directly. Looking at `hook-utils.sh:53-57`, `warn()` calls `exit 0` immediately after the echo. The plan's recommendation to use `echo >&2` directly (the "warn-then-continue" pattern) is correct since the marker check is placed BEFORE the final `allow` and the postcondition should continue to the final success message and `allow`. The plan already identifies this correctly in Phase 2, section 2 (lines 154-159).

### 2. get_field dependency for issue_number in artifact-comment-validator.sh

The plan proposes `issue_number=$(get_field '.tool_input.number')` in `artifact-comment-validator.sh`. The `create_comment` tool input uses `number` as the issue identifier. This is correct -- `ralph_hero__create_comment` receives `number` in its tool input, matching the `get_field` call.

### 3. Plan frontmatter shows status: draft

The plan frontmatter at line 3 says `status: draft`. This is expected for a plan in "Plan in Review" state -- it will be updated to `complete` after review approval. Not an issue.

### 4. Phase 2 code placement in artifact-comment-validator.sh could be more precise

The plan provides two slightly different approaches for where to place the marker write logic in `artifact-comment-validator.sh` -- inside the loop vs. after the loop. The "Final structure" section (lines 132-137) resolves this by placing it AFTER the loop (step 4) and BEFORE `allow` (step 5). The implementation note on line 130 clarifies that the cleanest placement is "just before allow at line 52." This is clear enough for the implementer.
