# Plan Critique: GH-512 -- pipeline_dashboard Per-Iteration Breakdown

## Verdict: APPROVED

Phase 3 of the iteration field support group plan is well-structured, technically accurate, and ready for implementation.

## Dimension Analysis

### 1. Completeness

**PASS**. Phase 3 defines 3 concrete changes across 2 files:

1. **GraphQL fragment addition** to `DASHBOARD_ITEMS_QUERY` in `dashboard-tools.ts` -- the query at lines 197-241 currently only fragments on `ProjectV2ItemFieldSingleSelectValue`; adding the `ProjectV2ItemFieldIterationValue` fragment is straightforward.
2. **`toDashboardItems()` extension** in `dashboard-tools.ts` -- the conversion function at lines 161-191 currently extracts only `workflowState`, `priority`, and `estimate` via the `getFieldValue()` helper. The plan correctly identifies that iteration data extraction should happen inline in `toDashboardItems()` rather than through `getFieldValue()` (since the iteration shape differs from single-select).
3. **`buildIterationSection()` in `dashboard.ts`** -- a new pure function that groups items by `iterationTitle`, calls the existing `aggregateByPhase()` on each subset, and appends a section to dashboard output.

Each change has a clear file target, line references, and code snippets.

### 2. Feasibility

**PASS**. All referenced files and code locations verified:

- `dashboard-tools.ts` exists with `DASHBOARD_ITEMS_QUERY` at lines 197-241
- `toDashboardItems()` exists at lines 161-191 with the expected structure
- `getFieldValue()` at lines 144-154 correctly only handles single-select (confirming plan's approach)
- `dashboard.ts` exists with `DashboardItem` at lines 30-43 (note: the plan parenthetically says `types.ts` in change #2, but the actual location is `dashboard.ts` -- minor inaccuracy, trivially resolved)
- `buildDashboard()` at lines 614-721 provides the exact pattern for adding per-iteration breakdowns (matching per-project and per-repo breakdowns)
- `RawDashboardItem` at lines 121-142 would need fields added for iteration node data (not explicitly called out but discoverable)

The existing per-project/per-repo breakdown pattern in `buildDashboard()` provides a clear template to follow. The guard clause (`items.some(i => i.iterationId)`) ensures zero regression for projects without iteration fields.

### 3. Clarity

**PASS**. Three success criteria, all specific and testable:

- Automated: Dashboard includes `## Sprint 1` section with phase counts when items have iteration assignments
- Automated: Projects without iteration field produce identical output (no regression)
- Manual: `pipeline_dashboard` shows per-sprint breakdown when issues are assigned to sprints

The automated criteria map directly to new test cases in `dashboard.test.ts`.

### 4. Scope

**PASS**. The overall plan's exclusions are well-defined. Phase 3 stays tightly scoped to read-only dashboard aggregation -- no mutation paths, no sprint lifecycle management.

## Minor Notes (Non-Blocking)

1. **DashboardItem location**: Change #2 parenthetically references `types.ts` for adding `iterationId`/`iterationTitle` to `DashboardItem`, but the type is actually defined in `dashboard.ts` (line 30). Implementer will discover this immediately.

2. **RawDashboardItem extension**: The plan does not explicitly mention extending the `RawDashboardItem` interface (lines 121-142 in `dashboard-tools.ts`) to accommodate iteration fields in the `fieldValues.nodes` type. The implementer will need to add `iterationId?`, `title?`, `startDate?`, `duration?` to the node union type. This is implicit from adding the GraphQL fragment.

3. **Formatter updates**: The plan mentions output format for the iteration section but does not explicitly detail updates to `formatMarkdown()` and `formatAscii()` in `dashboard.ts`. The implementer should add iteration rendering to both formatters, following the per-project/per-repo pattern. This is a minor gap -- the existing pattern is clear enough to follow.

## Dependency Validation

Phase 3 depends on Phase 1 (#508) for:
- `ProjectV2ItemFieldIterationValue` type with `iterationId` field (currently missing from type at types.ts:106-112)
- Iteration metadata shape (title, startDate, duration) -- already present in current type

Phase 3 does NOT need `resolveIterationId()` or `FieldOptionCache.getIterations()` since it reads iteration data directly from GraphQL responses. The dependency is correctly minimal.

## Conclusion

The plan is complete, feasible, clear, and well-scoped. The minor notes above are all trivially resolvable during implementation and do not warrant returning the plan for iteration. Approved for implementation.
